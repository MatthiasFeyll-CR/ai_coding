## System Overview

Ralph Pipeline orchestrates multi-milestone software projects using AI agents. The Ralph coding agent (Phase 2) is driven by an iterative loop:

1. Each iteration reads `.ralph/CLAUDE.md` as the prompt
2. CLAUDE.md tells Ralph what workflow to follow
3. Ralph reads `.ralph/context.md` for domain knowledge
4. Ralph picks a story, implements, tests, commits, updates progress

`.ralph/CLAUDE.md` is written **once** by the Pipeline Configurator during the planning phase, before any execution happens.

---

## The Problem

CLAUDE.md is a static file created before Phase 0 bootstrap and before any milestone runs. It cannot adapt to runtime reality.

### What CLAUDE.md contains (from Pipeline Configurator skill)

From `pipeline_configurator/SKILL.md`:

```
What CLAUDE.md contains:
1. Workflow process — Read prd.json, pick story, implement, test, commit
2. Read order — context.md first, then prd.json, then progress.txt
3. Completion signal — <promise>COMPLETE</promise>
4. Self-protection note — "Do NOT modify .ralph/CLAUDE.md"

What CLAUDE.md does NOT contain:
- Project-specific commands
- Test instructions
- Technology references
```

### What CLAUDE.md can't know

| Runtime information | Why it matters | Where it actually lives |
|--------------------|----|-----|
| Test commands generated by Phase 0 | Ralph needs to run tests after implementing | Written to `pipeline-config.json` by Phase 0 — CLAUDE.md was created before Phase 0 |
| Build commands | Ralph may need to build before testing | Same — Phase 0 generates these |
| Service setup commands | Database/Redis must be running for tests | Same — Phase 0 generates these |
| Deviations from prior milestones | Architecture may have changed | `.ralph/archive/*/progress.txt` and reconciliation docs |
| Current milestone's context quality | Whether context.md has all needed sections | Only knowable after Phase 1 runs |

### The design intent vs. reality

The design says: "CLAUDE.md is process-only; context.md carries domain knowledge."

But **test commands and quality check commands are process**, not domain. Ralph needs to know HOW to test, not just WHAT to test. The Pipeline Configurator writes CLAUDE.md before Phase 0 generates the actual test commands. So:

- CLAUDE.md says "run quality checks" but doesn't specify the commands
- context.md may or may not include the commands (depends on AI assembly)
- Ralph must extract test commands from an unstructured markdown document

### Ralph loop code

In `src/ralph_pipeline/phases/ralph_execution.py`:

```python
def _run_ralph_loop(...) -> bool:
    claude_md = scripts_dir / "CLAUDE.md"
    if not claude_md.exists():
        plogger.warning(
            f"CLAUDE.md not found at {claude_md} — Ralph has no instructions to execute"
        )
        return False

    for iteration in range(1, max_iterations + 1):
        prompt = claude_md.read_text()  # Same static file every iteration
        result = claude.run(prompt=prompt, ...)
```

Every iteration reads the exact same CLAUDE.md. No dynamic adaptation. No injection of runtime state.

## Impact

- **Severity:** Medium — mitigated by context.md if the PRD Writer includes quality commands, but that's not guaranteed
- **Failure mode:** Ralph proceeds without knowing test commands; all quality feedback deferred to QA phase
- **Frequency:** Every milestone — CLAUDE.md is static across the entire pipeline run
- **Blast radius:** Ralph's per-iteration test-fix loop is ineffective if it can't find the test commands

---

## Questions to Consider

1. Should CLAUDE.md be regenerated per-milestone by the pipeline code (not the AI), injecting current quality check commands?
2. Should the pipeline append a structured footer to CLAUDE.md before each milestone with test commands from `pipeline-config.json`?
3. Should CLAUDE.md be a Jinja-style template that gets rendered with runtime values?
4. Is the strict "CLAUDE.md = process only" principle correct? Should test commands be in CLAUDE.md or context.md?
5. Should the Ralph loop inject additional state (iteration count, which stories are done, current test command) into the prompt alongside CLAUDE.md?
6. Should CLAUDE.md reference `pipeline-config.json` directly and instruct Ralph to read quality commands from there?
